**Краткий ответ:**
Убить процесс — это значит послать ему **сигнал**, который говорит ему завершить работу. Разные сигналы ведут себя по-разному.

*   **Основные команды:** `kill [PID]`, `pkill [имя]`, `killall [имя]`.
*   **Правильный сигнал — `SIGTERM`** (вежливый запрос на завершение). 
*   **Сигнал `SIGKILL` (`kill -9`)** —  не дает процессу шанса на корректное завершение и может привести к потере данных.
*   **Идеальный сценарий:** Найти PID -> послать `SIGTERM` -> подождать -> если не помогло, послать `SIGKILL`.

### Основные команды

Есть три главные команды для "убийства" процессов: `kill`, `pkill` и `killall`.

#### 1. `kill` (работает по PID)

Используется, когда вы знаете **ID процесса (PID)**.

```bash
# Послать сигнал SIGTERM (вежливое завершение) процессу с PID 1234
kill 1234

# То же самое, что и выше (SIGTERM - сигнал по умолчанию)
kill -TERM 1234

# Послать сигнал SIGKILL (принудительное завершение) процессу с PID 1234
kill -KILL 1234

# Или используя номер сигнала (9)
kill -9 1234
```

**Как найти PID?**
*   `ps aux | grep <имя_процесса>` — классический способ.
*   `pidof <имя_процесса>` — выводит только PID.
*   `top` или `htop` — интерактивные мониторы процессов.

#### 2. `pkill` (работает по имени)

Удобная команда, которая ищет процессы по **имени** и сразу посылает им сигнал.

```bash
# Вежливо завершить все процессы с именем "firefox"
pkill firefox

# Принудительно завершить все процессы с именем "chrome"
pkill -9 chrome

# Завершить процессы, чье имя точно совпадает с "myapp"
pkill -x myapp
```

#### 3. `killall` (аналогично pkill, но с дополнительными опциями)

Похож на `pkill`, но имеет немного другой синтаксис и опции.

```bash
# Вежливо завершить все процессы с именем "nginx"
killall nginx

# Принудительно завершить все процессы с именем "python"
killall -9 python

# Интерактивный режим: запрашивает подтверждение для каждого процесса
killall -i vim
```

### Главные сигналы для завершения процессов

Сигналы — это то, *как* именно мы просим процесс завершиться. Вот самые важные из них:

| Сигнал      | Номер  | Команда для отправки                   | Что делает?                                                                                                                                                                                                                         |
| :---------- | :----- | :------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **SIGTERM** | **15** | `kill -TERM <PID>` или `kill <PID>`    | **Вежливая просьба завершиться.** Процесс может перехватить этот сигнал, чтобы корректно закрыть файлы, сохранить данные и завершить работу. Это **предпочтительный способ**.                                                       |
| **SIGKILL** | **9**  | `kill -KILL <PID>` или `kill -9 <PID>` | **Немедленное убийство.** Сигнал **нельзя перехватить или проигнорировать**. Ядро ОС сразу же забирает все ресурсы и уничтожает процесс. Рискует повредить данные. Это **крайняя мера**.                                            |
| **SIGHUP**  | 1      | `kill -HUP <PID>`                      | **"Повесить трубку"** (Hang Up). Изначально сигнал посылался при обрыве терминального соединения. Сейчас часто используется для просьбы **перезагрузить конфигурацию** без полного завершения работы (например, `nginx -s reload`). |
| **SIGINT**  | 2      | Нажать `Ctrl+C` в терминале            | **Прерывание (Interrupt).** Посылается при нажатии `Ctrl+C` в терминале. Просит процесс мягко завершиться.                                                                                                                          |

### Особый случай: Зависший процесс, который не отвечает

Иногда `SIGTERM` и `SIGKILL` не срабатывают. Это может произойти, если процесс находится в состоянии **`D` (Uninterruptible sleep)** — обычно из-за ожидания ввода/вывода от неотвечающего hardware (например, сломанный NFS-сервер). В таком состоянии процесс **игнорирует любые сигналы**, даже `SIGKILL`.

**Что делать?**
Остается только одно: перезагрузить систему или дождаться, пока hardware-проблема не решится сама собой (что может никогда не произойти).
