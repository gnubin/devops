Идеальный вопрос! Это современный и очень востребованный стек технологий. Сбор логов и метрик в Kubernetes имеет свою специфику из-за его динамической природы. Вот подробное объяснение, как это устроено.

### Общая архитектура стека (TL;DR)

Классический и самый популярный стек выглядит так:

1.  **Для метрик:** **Prometheus** собирает метрики со всех компонентов кластера и приложений.
2.  **Для логов:** **Fluent Bit** (или Fluentd) собирает логи с нод и подает их в **Loki**.
3.  **Для визуализации:** **Grafana** подключается и к Prometheus, и к Loki, чтобы отображать всё в едином интерфейсе.

Этот стек часто называют **PLG** (Prometheus-Loki-Grafana) или **FPLG** (Fluent Bit - Prometheus - Loki - Grafana).

---

### Детальное описание процесса

#### Часть 1: Сбор МЕТРИК (Monitoring)

**Главный инструмент: Prometheus**

**Задача:** Собирать числовые данные о производительности (загрузка CPU, потребление памяти, количество HTTP-запросов, latency и т.д.).

**Как это работает в Kubernetes:**

1.  **Prometheus Server** устанавливается в кластер (обычно через Helm chart или оператор). Он является центральным компонентом, который "вытягивает" (scrape) метрики по HTTP.
2.  **Источники метрик:**
    *   **Kubernetes Components:** API-сервер, ноды, kubelet, kube-proxy и т.д. сами предоставляют метрики на эндпоинтах (`/metrics`).
    *   **Приложения:** Ваши микросервисы должны также предоставлять метрики на эндпоинте `/metrics` (для этого используются библиотеки like Prometheus Client для Python, Go, Java и др.).
    *   **Exporters:** Специальные sidecar-контейнеры, которые преобразуют метрики из формата, непонятного Prometheus, в понятный. Например:
        *   `node-exporter` — собирает метрики по Linux-хосту (ноде).
        *   `redis-exporter` — собирает метрики из Redis.
3.  **Обнаружение целей (Service Discovery):** Kuberenetes постоянно меняется (поды появляются и исчезают). Prometheus умеет автоматически обращаться к Kubernetes API, обнаруживать все цели для сбора метрик (поды, сервисы) и начинать их опрос. Это его ключевая фича.

![Архитектура сбора метрик](https://grafana.com/static/img/docs/grafana-cloud/kubernetes-monitoring-architecture.png)

**Итог:** Prometheus автоматически находит всё, что можно мониторить, и хранит метрики в своем временном рядовом хранилище.

#### Часть 2: Сбор ЛОГОВ (Logging)

**Главные инструменты: Fluent Bit / Fluentd + Loki**

**Задача:** Собирать текстовые логи (stdout/stderr контейнеров, логи нод).

**Как это работает в Kubernetes:**

1.  **Сбор на узле (DaemonSet):** На каждой ноде кластера запускается легковесный агент (чаще всего **Fluent Bit**) в виде **DaemonSet**. Его задача — читать логи всех контейнеров, которые работают на этой ноде. Логи контейнеров по умолчанию пишутся в файлы на ноде (`/var/log/containers/*.log`).
2.  **Обработка и обогащение:** Fluent Bit может парсить логи, добавлять к ним метаданные (например, `namespace`, `pod_name`, `container_name`, `labels`), что критически важно для их последующего поиска.
3.  **Отправка в хранилище:** Обработанные логи отправляются в **Loki**. Loki — это не как Elasticsearch. Он не индексирует *тело* лога, а только его **метки (labels)**. Это делает его невероятно экономичным и быстрым.
4.  **Loki** принимает логи, сжато их хранит и предоставляет API для запросов.

**Альтернатива (классическая, но более тяжелая):** Fluent Bit -> Kafka (как буфер) -> Fluentd (как агрегатор) -> Elasticsearch. Loki вытесняет этот стек благодаря простоте и эффективности.

![Архитектура сбора логов](https://grafana.com/static/img/docs/loki/v2/flow.png)

#### Часть 3: ВИЗУАЛИЗАЦИЯ и АНАЛИЗ (Grafana)

**Главный инструмент: Grafana**

**Задача:** Предоставить единый дашборд для метрик и логов.

**Как это работает:**

1.  **Добавление источников данных (Data Sources):** В Grafana добавляются два источника данных:
    *   **Prometheus** (для метрик)
    *   **Loki** (для логов)
2.  **Создание дашбордов:**
    *   **Для метрик:** С помощью языка запросов **PromQL** вы создаете графики: загрузка CPU по нодам, количество 5xx ошибок от вашего ingress-контроллера, кастомные метрики бизнес-логики.
    *   **Для логов:** С помощью языка запросов **LogQL** вы можете искать и фильтровать логи. Самая мощная фича — **"переход от метрики к логу"**. Вы можете кликнуть по пику на графике ошибок и сразу же **"провалиться" (drill down)** в соответствующие логи за этот период времени прямо в интерфейсе Grafana. Это радикально ускоряет расследование инцидентов.

### Практическое развертывание (как это сделать)

Чаще всего весь этот стек разворачивается с помощью **Helm** (менеджер пакетов для Kubernetes).

1.  Установить **Prometheus Stack** (часто называется `kube-prometheus-stack`). Этот один Helm-чарт устанавливает:
    *   Prometheus Server
    *   Огромное количество exporters (node-exporter и др.)
    *   Alertmanager (для оповещений)
    *   И чаще всего **саму Grafana** с предустановленными дашбордами.

    ```bash
    helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
    helm install prometheus prometheus-community/kube-prometheus-stack
    ```

2.  Установить **Loki Stack** (включает Loki и агент Fluent Bit):

    ```bash
    helm repo add grafana https://grafana.github.io/helm-charts
    helm install loki grafana/loki-stack --set fluent-bit.enabled=true
    ```

3.  Настроить Grafana на использование Loki в качестве источника данных (часто это уже сделано автоматически чартами).

4.  Создавать свои дашборды или импортировать готовые из огромной библиотеки [Grafana Dashboards](https://grafana.com/grafana/dashboards/).

### Итог для собеседования

"Для сбора метрик и логов в Kubernetes мы используем современный стек на основе **Prometheus** и **Loki**, с визуализацией в **Grafana**.

1.  **Метрики:** **Prometheus** автоматически обнаруживает и собирает метрики со всех компонентов кластера и приложений через свое **Service Discovery**.
2.  **Логи:** На каждой ноде в виде **DaemonSet** работает легковесный агент **Fluent Bit**, который собирает логи всех подов и отправляет их в **Loki**. Loki хранит логи эффективно, индексируя только метки, а не их содержимое.
3.  **Визуализация:** **Grafana** подключается к Prometheus и Loki как к источникам данных. Это позволяет не только строить дашборды, но и реализовывать мощный сценарий **"от метрики к логу"**, когда мы можем кликнуть на аномалию на графике и мгновенно увидеть логи, которые к ней привели.

Весь этот стек обычно разворачивается с помощью **Helm-чартов**, что делает его установку и управление стандартизированным и воспроизводимым процессом. Этот подход стал де-факто стандартом для мониторинга Kubernetes-кластеров."
