### Введение (контекст)

"В моей практике настройка параметров ядра для ВМ чаще всего была связана с **высоконагруженными хостами виртуализации** под управлением KVM и **контейнерами в продакшене**. Цели были разные: кому-то нужна была максимальная пропускная способность сети, кому-то — низкая задержка, а для баз данных — предсказуемая дисковая IOPS. Ядро — это посредник между железом и гостевыми ОС, и его настройка критически важна для эффективности."

### Ключевые направления настройки и конкретные параметры

Я бы разделил свой опыт на несколько ключевых областей.

#### 1. Оптимизация ввода-вывода (I/O) для дисковых операций

Это часто самое "узкое место".

*   **Выбор и настройка планировщика I/O (I/O Scheduler):**
*   **Проблема:** Старый планировщик `cfq` (Completely Fair Queuing) был справедливым, но не самым производительным для быстрых SSD/NVMe в виртуальной среде.
*   **Решение:** Переход на `mq-deadline` или `kyber` для SSD/NVMe. Для задач, чувствительных к задержкам, `mq-deadline` обеспечивает более предсказуемое время отклика, чем `kyber`.
*   **Действие:** Установка планировщика для всех дисков (например, NVMe) в `/etc/udev/rules.d/60-ioscheduler.rules`:
```bash
ACTION=="add|change", KERNEL=="nvme[0-9]n[0-9]", ATTR{queue/scheduler}="mq-deadline"
```

*   **Увеличение лимитов очереди (Queue Depth):**
*   **Проблема:** Виртуальные машины конкурируют за доступ к физическому диску. Недостаточная глубина очереди (`nr_requests`) на хосте может лимитировать общую производительность.
*   **Решение:** Увеличение глубины очереди для SSD-дисков, чтобы лучше параллелить операции.
*   **Действие:** Тот же `udev` rule:
```bash
ACTION=="add|change", KERNEL=="sd[a-z]|nvme[0-9]n[0-9]", ATTR{queue/nr_requests}="2048"
```

#### 2. Сетевая производительность

Цель — уменьшить задержку (latency) и нагрузку на CPU, увеличить пропускную способность (throughput).

*   **Включение многочековой обработки (RSS) и отключение ненужных функций:**
*   **Проблема:** Один сетевой интерфейс обрабатывается одним CPU-ядром, создавая bottleneck.
*   **Решение:** Включение `RSS` (Receive Side Scaling) на сетевой карте и настройка `irqbalance` для правильного распределения прерываний (IRQ) по ядрам CPU.
*   **Действие:** Проверка и настройка драйвера сетевой карты (например, `ethtool -L eth0 combined 16`).

*   **Настройка параметров ядра для обработки пакетов:**
*   **`net.core.netdev_max_backlog`**: Увеличивается, чтобы избежать потерь пакетов при резких всплесках входящего трафика.
*   **`net.core.rmem_max` / `net.core.wmem_max`**: Увеличиваются буферы для приема и отправки данных, чтобы "прокачать" больше данных за один раз, особенно важно для 10G/40G сетей.
*   **`net.ipv4.tcp_rmem` / `net.ipv4.tcp_wmem`**: Настройка автонастройки TCP-буферов для эффективного использования всей доступной пропускной способности (Bandwidth-Delay Product).
*   **`net.ipv4.tcp_congestion_control = bbr`**: Установка современного алгоритма контроля перегрузки от Google, который часто показывает лучшую производительность на длинных каналах с потерей пакетов.

#### 3. Управление памятью (Memory Management)

*   **Транспарентная огромная страница (Transparent HugePages - THP):**
*   **Проблема:** THP (`defrag`) может вызывать латентность из-за работы `khugepaged` в фоновом режиме, что очень плохо для workloads с низкой задержкой (например, базы данных like Redis).
*   **Решение:** **Отключение** `defrag` для хоста базы данных или установка в `madvise`.
*   **Действие:** `echo madvise > /sys/kernel/mm/transparent_hugepage/defrag`
*   *Важно:* Для обычных веб-приложений THP обычно полезны, так как уменьшают TLB misses.

*   **Swappiness:**
*   **Проблема:** Ядро слишком агрессивно сбрасывает страницы памяти на swap, даже когда физическая память еще не исчерпана. Это создает ненужные дисковые I/O и замедляет работу.
*   **Решение:** Уменьшение значения `vm.swappiness` (например, до `10` или даже `1`). На серверах с большим объемом RAM swap часто нужен только как аварийный буфер, а не как активный участник управления памятью.
*   **Действие:** `sysctl -w vm.swappiness=10`

#### 4. Виртуализация (модуль KVM)

*   **Выбор механизма виртуализации CPU:**
*   **Проблема:** Старый `kvm-intel` может не использовать все возможности современного CPU.
*   **Решение:** Включение вложенной виртуализации (nested VT) если нужно, и проверка что используется максимально эффективный режим (например, `host-passthrough` или `host-model` в libvirt), чтобы гость видел реальные CPU-флаги хоста, а не общий набор.

*   **Многопоточная обработка vhost-net:**
*   **Проблема:** Сетевой трафик виртуальных машин обрабатывается одним ядром хоста.
*   **Решение:** Включение `vhost_net` модуля и настройка количества его потоков.
*   **Действие:** `modprobe vhost_net threads=2`

### Заключение (философия подхода)

"Важно понимать, что **не существует волшебного набора параметров**. Моя работа всегда начиналась с профилирования:
1.  **Мониторинг:** Использование `perf`, `iostat -x 1`, `vmstat 1`, `sar` для выявления реальных bottleneck'ов: высокий `iowait`, softirq, нехватка памяти.
2.  **Тестирование:** Нагрузочное тестирование до и после каждого изменения с помощью `fio` (диски), `iperf3` (сеть), `sysbench` (CPU, память).
3.  **Документирование:** Все изменения вносились системно (через `sysctl.d/`, `udev rules`, `modprobe.d/`) и тщательно документировались. Мы никогда не меняли параметры "наугад" и всегда могли откатиться.
4.  **Приоритизация:** Сначала лечились самые большие боли, выявленные профилированием. Настройка "на всякий случай" часто бесполезна и может быть вредна.

Например, для хоста с десятками ВМ, обслуживающих веб-трафик, приоритетом была сеть и CPU. Для хоста с 3-4 тяжелыми ВМ СУБД — дисковая подсистема и правильная настройка памяти."
