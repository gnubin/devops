### Основные компоненты GitLab CI/CD

GitLab CI/CD построен на основе конфигурационного файла и нескольких ключевых сущностей:

1.  **`.gitlab-ci.yml`** — **Сердце системы**. YAML-файл, который лежит в корне репозитория и описывает весь процесс CI/CD. В нем определяются этапы, задания и их поведение.
2.  **Pipeline (Пайплайн)** — Один полный цикл выполнения, который запускается по событию (коммит, мерж, по расписанию). Состоит из набора стадий и заданий.
3.  **Stages (Стадии)** — Последовательные этапы в пайплайне. Задания внутри одной стадии выполняются параллельно (если возможно), а сами стадии выполняются последовательно. Пример: `.test` -> `.build` -> `.deploy`.
4.  **Jobs (Задания)** — Это отдельные "шаги" или "единицы работы". Это самая маленькая сущность, которая выполняет скрипты. Каждое задание относится к одной стадии.
    *   `script` — что выполнить.
    *   `rules`/`only`/`except` — когда выполнять.
5.  **Runner (Исполнитель)** — Агент, который выполняет задания. Это может быть заранее подготовленный shell-раннер на конкретной машине или более гибкий и изолированный **Docker-раннер**, который мы использовали чаще всего.
6.  **Artifacts (Артефакты)** — Файлы, которые создаются в процессе выполнения задания (например, собранный JAR-файл, отчеты о тестировании) и передаются между заданиями.
7.  **Cache (Кэш)** — Механизм для повторного использования неделимых данных между запусками пайплайна (например, кэширование зависимостей `node_modules` или `vendor/`).
8.  **Environments (Окружения)** и **Deployments** — Позволяют отслеживать развертывания в разные среды (например, `staging`, `production`). Показывают, какая версия кода сейчас на каком стенде.
9.  **Variables (Переменные)** — Ключевой механизм для параметризации пайплайнов. Бывают:
    *   **Predefined:** `CI_COMMIT_SHA`, `CI_JOB_STATUS` (автоматически предоставляются GitLab).
    *   **Custom:** Задаются в настройках проекта/группы для хранения секретов (токены, пароли) или общих настроек.
10. **Merge Request Pipelines:** Отдельные пайплайны, которые запускаются для веток и показывают свои статусы прямо в интерфейсе Merge Request. Это основа практик Code Review.

### Лучшие практики, которые я применял

1.  **Идемпотентность пайплайнов:** Пайплайн должен давать одинаковый результат при повторном запуске с одними и теми же входными данными. Это достигается за счет чистых Docker-образов для раннеров и точной фиксации версий зависимостей.

2.  **Использование `rules` вместо `only`/`except`:**
    Новый синтаксис `rules` гораздо более гибкий и мощный.
    ```yaml
    deploy_to_prod:
      stage: deploy
      script: [...]
      rules:
        - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH # Если коммит в main
          when: manual # Запускать вручную
        - when: never # Для всех остальных случаев не запускать
    ```

3.  **Шаблонизация и скрытие сложности с помощью `extends` и `!reference`:**
    Чтобы избежать копипасты и сделать файл читаемым.
    ```yaml
    .base_test: &base_test
      image: python:3.9
      before_script:
        - pip install -r requirements.txt

    test_unit:
      <<: *base_test
      script:
        - pytest tests/unit/

    test_integration:
      <<: *base_test
      script:
        - pytest tests/integration/
    ```

4.  **Безопасность:**
    *   **Никогда не хранить секреты в `.gitlab-ci.yml`.** Все токены, пароли, SSH-ключи хранить в **Masked Variables** в настройках проекта/группы.
    *   Использовать **Docker-in-Docker (dind)** с проверенными образами для сборки своих образов.
    *   Сканировать образы на уязвимости (Trivy, Grype) в отдельном джобе.

5.  **Эффективное кэширование:**
    *   Кэшировать зависимости (например, `pip`, `npm`, `maven`), но использовать уникальный ключ, чтобы не повредить кэш из другой ветки.
    ```yaml
    cache:
      key: ${CI_COMMIT_REF_SLUG} # Ключ по имени ветки
      paths:
        - node_modules/
    ```

6.  **Оптимизация скорости:**
    *   Разбивать пайплайн на параллельно выполняемые задания (например, юнит-тесты и линтеры могут работать одновременно).
    *   Использовать маленькие и быстрые Docker-образы для раннеров (`alpine`-версии).
    *   Выносить долгие задачи (например, компиляцию) в отдельные этапы и запускать их только для определенных веток.

7.  **"Детерминированные" пайплайны:** Пайплайн должен четко отражать процесс разработки. Стадия тестирования не должна падать из-за проблем на стадии деплоя. Для этого используются `needs` для построения графа зависимостей вместо жесткой последовательности стадий.

8.  **Инкременментальная сборка и деплой:** Не собирать весь проект с нуля каждый раз. Использовать артефакты и кэш для ускорения.

Как в GitLab CI можно передать значение переменной из одного задания (job) в другое_

Вот основные методы, от самых простых и популярных к более сложным.

---

### 1. Артефакты (Artifacts) — Самый частый и правильный способ

**Для чего:** Чтобы передать **файлы** из одного задания в последующие. Например, собранный бинарник, отчет о тестировании, файл конфигурации, сгенерированный скриптом.

**Как работает:**
*   В job-"отправителе" вы определяете путь к файлам, которые нужно сохранить как артефакты.
*   GitLab сохранит эти файлы и сделает их доступными для скачивания в job-"получателе".
*   В получателе файлы будут лежать в той же самой директории, что и в отправителе.

**Пример:** Job `build` создает файл `version.txt`, а job `deploy` использует его.

```yaml
stages:
  - build
  - deploy

build_job:
  stage: build
  script:
    - echo "1.0.${CI_PIPELINE_ID}" > version.txt # Создаем файл
  artifacts:
    paths:
      - version.txt # Указываем, что этот файл - артефакт
    expire_in: 1 hour # Необязательно: время хранения артефакта

deploy_job:
  stage: deploy
  script:
    - cat version.txt # Файл version.txt уже здесь!
    - echo "Deploying version $(cat version.txt)"
  dependencies:
    - build_job # Явно указываем, от какого job нам нужны артефакты
```

**Важно:** По умолчанию артефакты из всех jobs предыдущих стадий передаются автоматически. Ключ `dependencies` позволяет точечно указать, артефакты от каких jobs нужны, если вы хотите ограничить передачу.

### 2. Кэш (Cache) — Для зависимостей и промежуточных результатов

**Для чего:** Чтобы **переиспользовать** неделимые данные между заданиями и даже между пайплайнами. Идеально для кэширования зависимостей (например, `node_modules/`, `vendor/` в Python/Go), чтобы не скачивать их каждый раз.

**Внимание:** Кэш не гарантирует целостность и актуальность данных. Это общее хранилище, которое может быть перезаписано из другой ветки или другого пайплайна.

**Пример:** Кэшируем зависимости Node.js между jobs.

```yaml
cache: # Глобальный кэш (можно определить и на уровне job)
  key: ${CI_COMMIT_REF_SLUG} # Ключ кэша (по ветке)
  paths:
    - node_modules/
    - .yarn-cache/

install_deps:
  stage: build
  script:
    - yarn install --frozen-lockfile --cache-folder .yarn-cache

build_app:
  stage: build
  script:
    - yarn build
  # Эта job автоматически получит доступ к закэшированной папке node_modules/
```

### 3. dotenv-артефакты — Передача переменных окружения

**Это самый мощный способ для передачи именно переменных.**

**Для чего:** Чтобы job-"отправитель" мог динамически создать значение переменной и передать его в виде переменной окружения в job-"получатель".

**Как работает:**
1.  В job-"отправителе" вы создаете файл с расширением `.env`.
2.  В этом файле вы определяете переменные в формате `KEY=VALUE`.
3.  Вы объявляете этот файл как артефакт специального типа `dotenv`.
4.  GitLab автоматически подхватит переменные из этого файла и сделает их переменными окружения для всех последующих jobs.

**Пример:** Job определяет динамическую версию сборки, а job деплоя ее использует.

```yaml
stages:
  - prepare
  - deploy

set_version_job:
  stage: prepare
  script:
    # Динамически вычисляем версию и записываем в .env-файл
    - echo "CUSTOM_VERSION=1.0.${CI_PIPELINE_ID}" > build.env
  artifacts:
    reports:
      dotenv: build.env # Ключевой момент! Объявляем dotenv-артефакт.

deploy_job:
  stage: deploy
  script:
    # Теперь переменная $CUSTOM_VERSION доступна здесь!
    - echo "Deploying version $CUSTOM_VERSION"
    - scp myapp-$CUSTOM_VERSION.zip user@server:/opt/app
```

### 4. Рукопожатие через API (наиболее гибкий, но сложный способ)

**Для чего:** Для передачи сложных структур данных (например, JSON) или для передачи данных между пайплайнами.

**Как работает:**
1.  Job-"отправитель" с помощью скрипта и **API GitLab** обновляет переменную на уровне пайплайна или проекта.
2.  Job-"получатель" просто использует эту переменную.

**Пример:** Обновление переменной пайплайна через API.

```yaml
stages:
  - prepare
  - use

set_var_job:
  stage: prepare
  script:
    # Используем API GitLab, чтобы обновить переменную текущего пайплайна
    - 'curl --request PUT --header "PRIVATE-TOKEN: $MY_ACCESS_TOKEN" --form "variable=MY_VAR" --form "value=my_value" "${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/variables/MY_VAR"'

use_var_job:
  stage: use
  script:
    - echo $MY_VAR # Переменная доступна, так как она установлена на уровне пайплайна/проекта
```

**Недостаток:** Нужно управлять токенами доступа (`$MY_ACCESS_TOKEN`) с соответствующими правами.

### Сводная таблица выбора метода

| Метод | Что передаем | Когда использовать |
| :--- | :--- | :--- |
| **Артефакты (artifacts)** | Файлы, директории | Собранные бинарники, отчеты, конфиги. |
| **dotenv-артефакты** | **Переменные окружения** | **Динамически вычисленные значения для других jobs (идеальный ответ на ваш вопрос).** |
| **Кэш (cache)** | Зависимости, промежуточные результаты | `node_modules`, `vendor/bundle`, чтобы ускорить следующие запуски. |
| **API GitLab** | Что угодно | Сложные сценарии, передача данных между пайплайнами. |

### Итог для собеседования

"В GitLab CI есть несколько основных способов передать данные между заданиями:

1.  **Артефакты (`artifacts`)** — лучший способ для передачи **файлов** (собранный код, отчеты).
2.  **dotenv-артефакты (`reports: dotenv`)** — это **ключевой и самый удобный способ для передачи переменных**. Первое задание записывает переменные в файл, а GitLab автоматически подхватывает их и делает переменными окружения для последующих заданий.
3.  **Кэш (`cache`)** — для переиспользования зависимостей между заданиями и пайплайнами для экономии времени.
4.  **API GitLab** — наиболее гибкий метод для сложных сценариев, например, обновления переменных пайплайна программно.

Для передачи именно **значения переменной** из одного задания в другое я бы в первую очередь использовал механизм **dotenv-артефактов**, так как он предназначен именно для этой задачи и требует минимальных усилий для настройки."
