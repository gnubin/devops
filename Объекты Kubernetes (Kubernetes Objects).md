Конечно! Kubernetes оперирует множеством объектов (resources), которые являются абстракциями для описания желаемого состояния кластера. Вот практически полный список основных и популярных объектов, сгруппированных по категориям.

### 1. Рабочие нагрузки (Workloads)
*Объекты для развертывания и управления приложениями.*
*   **[[Pod]]:** Наименьшая и simplest вычислительная единица. "Обёртка" для одного или нескольких контейнеров.
*   **[[Deployment]]:** Наиболее распространённый объект для управления stateless-приложениями. Обеспечивает обновление, откат и масштабирование Pod'ов.
*   **[[ReplicaSet]]:** Обеспечивает работу заданного количества идентичных Pod'ов. Обычно управляется Deployment'ом.
*   **StatefulSet:** Для управления stateful-приложениями (БД, Kafka, Elasticsearch). Гарантирует упорядоченность развёртывания, уникальность и постоянство хранилища для каждого Pod'а.
*   **DaemonSet:** Гарантирует, что Pod будет работать на каждой (или на определённой) ноде кластера. Идеально для агентов мониторинга (node-exporter) или логгирования (fluentd).
*   **Job:** Создает один или несколько Pod'ов, которые выполняют задачу и завершают работу (с кодом 0).
*   **CronJob:** Управляет Jobs, которые запускаются по расписанию (как cron в Linux).

### 2. Сеть и сервисы (Networking & Service Discovery)
*Объекты для обеспечения сетевого взаимодействия.*
*   **[[Service]]:** Абстракция, предоставляющая стабильную точку доступа (IP/DNS) к набору динамических Pod'ов. Типы: `ClusterIP`, `NodePort`, `LoadBalancer`.
*   **[[Ingress]]:** Управляет внешним доступом к сервисам в кластере, обычно по HTTP/HTTPS. Обеспечивает маршрутизацию на основе хоста или пути.
*   **IngressClass:** Описывает тип Ingress-контроллера, который должен обрабатывать Ingress.
*   **NetworkPolicy:** "Файервол" для Pod'ов. Определяет правила, как группа Pod'ов может общаться друг с другом и другими сетевыми endpoint'ами.

### 3. Конфигурация и хранение (Configuration & Storage)
*Объекты для работы с конфигурацией, секретами и данными.*
*   **ConfigMap:** Хранит конфигурационные данные в виде пар "ключ-значение". Монтируется в Pod'ы как переменные окружения или файлы.
*   **Secret:** Похож на ConfigMap, но для хранения конфиденциальных данных (пароли, токены, ключи). Данные хранятся в base64-encoded виде.
*   **PersistentVolume (PV):** Представляет собой кусок сетевого хранилища в кластере (как ресурс).
*   **PersistentVolumeClaim (PVC):** Заявка на использование хранилища (запрашивает PV определённого размера и с определёнными правами доступа).
*   **StorageClass:** Описывает "класс" хранилища (например, "быстрое SSD", "медленное HDD"). Позволяет использовать Dynamic Provisioning.

### 4. Кластерные метаданные (Cluster Metadata)
*Объекты, описывающие сам кластер и его политики.*
*   **Namespace:** Виртуальный кластер внутри физического. Обеспечивает изоляцию ресурсов и их группировку (например, `dev`, `stage`, `prod`).
*   **ResourceQuota:** Ограничивает потребление ресурсов (CPU, RAM, количество объектов) within a Namespace.
*   **LimitRange:** Ограничивает ресурсы (CPU, RAM), которые могут быть запрошены или использованы одним Pod'ом или Контейнером within a Namespace.

### 5. Безопасность (Security)
*Объекты для управления доступом и правами (RBAC).*
*   **ServiceAccount:** Предоставляет идентификатор для Pod'ов, которые работают внутри кластера и взаимодействуют с API Server.
*   **Role:** Определяет набор правил доступа к ресурсам *within a specific Namespace*.
*   **ClusterRole:** То же, что Role, но для доступа к ресурсам *всего кластера* (например, к Nodes, PV) или к ресурсам across all namespaces.
*   **RoleBinding:** Привязывает Role или ClusterRole к пользователю, группе или ServiceAccount *within a specific Namespace*.
*   **ClusterRoleBinding:** Привязывает ClusterRole к пользователю, группе или ServiceAccount *на уровне всего кластера*.

### 6. Подкапотные / Системные (Underlying / System)
*Объекты, которые обычно создаются и управляются другими объектами или самой системой.*
*   **Endpoint:** Хранит IP-адреса всех Pod'ов, которые соответствуют селектору Service'а. Создается и поддерживается автоматически.
*   **Event:** Запись о событии, произошедшем с каким-либо объектом (например, "Pod scheduled", "Image pulled"). Просматривается через `kubectl get events`.
*   **Node:** Представление рабочей ноды (сервера) в кластере Kubernetes.

### 7. Расширения (Extensions)
*Объекты для расширения функциональности Kubernetes (CRD).*
*   **CustomResourceDefinition (CRD):** Позволяет определять свои собственные, кастомные объекты (Resources) в Kubernetes, расширяя его API. На основе CRD строятся операторы (Operators).

### Как посмотреть все объекты в вашем кластере?

1.  **Посмотреть все доступные типы объектов API:**
    ```bash
    kubectl api-resources
    ```
    Эта команда покажет список (имя, короткие алиасы, группа API,是不是 namespaced, kind).

2.  **Посмотреть объекты определенного типа:**
    ```bash
    kubectl get <resource-name>  # Например: kubectl get pods, kubectl get services
    kubectl get <shortname>      # Используя алиас: kubectl get po, kubectl get svc
    ```

Этот список охватывает ~95% объектов, с которыми вы будете сталкиваться на практике. Основа — это **Pod, Deployment, Service, ConfigMap, Secret, PersistentVolumeClaim и Namespace.**
