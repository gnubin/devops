### Мой опыт работы с GitLab CI

"В моей практике GitLab CI/CD был центральным инструментом для построения полного цикла доставки ПО (DevOps) в нескольких проектах. Мы использовали его для:

*   **Автоматизации тестирования:** Запуск юнит-тестов, линтеров, интеграционных и e2e-тестов на каждый коммит в merge request.
*   **Сборки артефактов:** Создание Docker-образов (и их сканирование на уязвимости), компиляция бинарников для разных платформ, сборка фронтенд-приложений.
*   **Развертывания (Deployment):** Реализация моделей **GitLab Environments** и **Review Apps** для автоматического развертывания feature-веток на тестовые стенды. Настройка деплоя в прод через мануальное задание (manual job) с подтверждением.
*   **Инфраструктура как код (IaC):** Запуск Terraform-план и применение изменений инфраструктуры через отдельный пайплайн.
*   **Мониторинг и оповещения:** Интеграция с Slack и Telegram для отправки уведомлений об успешных/неуспешных пайплайнах.

Ключевое преимущество, которое мы использовали — **единая платформа**. Не нужно настраивать интеграции между GitHub/Jenkins/Jira. Всё уже есть в одном месте: код, CI/CD, Issues, мониторинг пайплайнов."

### Основные компоненты GitLab CI/CD

GitLab CI/CD построен на основе конфигурационного файла и нескольких ключевых сущностей:

1.  **`.gitlab-ci.yml`** — **Сердце системы**. YAML-файл, который лежит в корне репозитория и описывает весь процесс CI/CD. В нем определяются этапы, задания и их поведение.
2.  **Pipeline (Пайплайн)** — Один полный цикл выполнения, который запускается по событию (коммит, мерж, по расписанию). Состоит из набора стадий и заданий.
3.  **Stages (Стадии)** — Последовательные этапы в пайплайне. Задания внутри одной стадии выполняются параллельно (если возможно), а сами стадии выполняются последовательно. Пример: `.test` -> `.build` -> `.deploy`.
4.  **Jobs (Задания)** — Это отдельные "шаги" или "единицы работы". Это самая маленькая сущность, которая выполняет скрипты. Каждое задание относится к одной стадии.
    *   `script` — что выполнить.
    *   `rules`/`only`/`except` — когда выполнять.
5.  **Runner (Исполнитель)** — Агент, который выполняет задания. Это может быть заранее подготовленный shell-раннер на конкретной машине или более гибкий и изолированный **Docker-раннер**, который мы использовали чаще всего.
6.  **Artifacts (Артефакты)** — Файлы, которые создаются в процессе выполнения задания (например, собранный JAR-файл, отчеты о тестировании) и передаются между заданиями.
7.  **Cache (Кэш)** — Механизм для повторного использования неделимых данных между запусками пайплайна (например, кэширование зависимостей `node_modules` или `vendor/`).
8.  **Environments (Окружения)** и **Deployments** — Позволяют отслеживать развертывания в разные среды (например, `staging`, `production`). Показывают, какая версия кода сейчас на каком стенде.
9.  **Variables (Переменные)** — Ключевой механизм для параметризации пайплайнов. Бывают:
    *   **Predefined:** `CI_COMMIT_SHA`, `CI_JOB_STATUS` (автоматически предоставляются GitLab).
    *   **Custom:** Задаются в настройках проекта/группы для хранения секретов (токены, пароли) или общих настроек.
10. **Merge Request Pipelines:** Отдельные пайплайны, которые запускаются для веток и показывают свои статусы прямо в интерфейсе Merge Request. Это основа практик Code Review.

### Лучшие практики, которые я применял

1.  **Идемпотентность пайплайнов:** Пайплайн должен давать одинаковый результат при повторном запуске с одними и теми же входными данными. Это достигается за счет чистых Docker-образов для раннеров и точной фиксации версий зависимостей.

2.  **Использование `rules` вместо `only`/`except`:**
    Новый синтаксис `rules` гораздо более гибкий и мощный.
    ```yaml
    deploy_to_prod:
      stage: deploy
      script: [...]
      rules:
        - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH # Если коммит в main
          when: manual # Запускать вручную
        - when: never # Для всех остальных случаев не запускать
    ```

3.  **Шаблонизация и скрытие сложности с помощью `extends` и `!reference`:**
    Чтобы избежать копипасты и сделать файл читаемым.
    ```yaml
    .base_test: &base_test
      image: python:3.9
      before_script:
        - pip install -r requirements.txt

    test_unit:
      <<: *base_test
      script:
        - pytest tests/unit/

    test_integration:
      <<: *base_test
      script:
        - pytest tests/integration/
    ```

4.  **Безопасность:**
    *   **Никогда не хранить секреты в `.gitlab-ci.yml`.** Все токены, пароли, SSH-ключи хранить в **Masked Variables** в настройках проекта/группы.
    *   Использовать **Docker-in-Docker (dind)** с проверенными образами для сборки своих образов.
    *   Сканировать образы на уязвимости (Trivy, Grype) в отдельном джобе.

5.  **Эффективное кэширование:**
    *   Кэшировать зависимости (например, `pip`, `npm`, `maven`), но использовать уникальный ключ, чтобы не повредить кэш из другой ветки.
    ```yaml
    cache:
      key: ${CI_COMMIT_REF_SLUG} # Ключ по имени ветки
      paths:
        - node_modules/
    ```

6.  **Оптимизация скорости:**
    *   Разбивать пайплайн на параллельно выполняемые задания (например, юнит-тесты и линтеры могут работать одновременно).
    *   Использовать маленькие и быстрые Docker-образы для раннеров (`alpine`-версии).
    *   Выносить долгие задачи (например, компиляцию) в отдельные этапы и запускать их только для определенных веток.

7.  **"Детерминированные" пайплайны:** Пайплайн должен четко отражать процесс разработки. Стадия тестирования не должна падать из-за проблем на стадии деплоя. Для этого используются `needs` для построения графа зависимостей вместо жесткой последовательности стадий.

8.  **Инкременментальная сборка и деплой:** Не собирать весь проект с нуля каждый раз. Использовать артефакты и кэш для ускорения.
