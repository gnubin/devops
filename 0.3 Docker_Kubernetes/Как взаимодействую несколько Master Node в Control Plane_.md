Control Plane (Управляющая плоскость) — это не одна "мастер-нода", а набор компонентов, которые для надежности и производительности почти всегда запускаются на нескольких отдельных серверах, которые и называются "мастер-нодами" (master nodes).
#### Почему одна мастер-нода — это плохо?
Когда у вас всего один сервер, на котором работает весь Control Plane (API Server, Scheduler и т.д.). Если этот сервер:
*   **Выключится** из-за сбоя питания.
*   **Упадет** из-за software bug.
*   **Потеряет сетевую связность.**
*   **Будет перегружен.**

то, весь ваш кластер Kubernetes перестанет функционировать. Вы не сможете управлять приложениями, а главное — кластер не сможет самовосстанавливаться (если упадет рабочий Pod, его никто не перезапустит) так как Master Node вышла из строя. 

Это называется **Single Point of Failure (SPOF)** — единая точка отказа. То есть если из строя выходит только один узел, вся система перестаёт корректно функционаровать.

#### Как это решается? High Availability (HA) Control Plane

Решение — запустить компоненты Control Plane на **нескольких независимых серверах (нодах)**, объединенных в кластер. Обычно используют нечетное количество мастер-нод: **3, 5 или больше**. Это нужно для механизма консенсуса в `etcd`.

**Как выглядит высокодоступный (HA) кластер:**

#### Как работают компоненты в HA-режиме?

*   Все **`etcd`** на разных мастер-нодах образуют **кластер**. Данные реплецируются между ними. Для любого изменения требуется консенсус (согласие) большинства нод (например, 2 из 3). Это гарантирует, что данные согласованы и надежны, даже если одна нода отвалится.

*   **`kube-apiserver`**: Каждый API Server работает **независимо** на своей мастер-ноде. Они все находятся перед одним общим кластером `etcd`. Обычно перед ними ставят **load balancer** (балансировщик нагрузки, например, HAProxy или cloud-решение). Пользователь и рабочие ноды обращаются не к конкретному API Server, а к IP-адресу этого load balancer'а, который распределяет запросы между всеми здоровыми API Server. Это обеспечивает **масштабируемость и отказоустойчивость**.

*   **`kube-scheduler`** и **`kube-controller-manager`**: Эти компоненты в каждый момент времени **активны только в одном экземпляре** на всей управляющей плоскости. Они используют механизм **лидерства** (`leader election`).
    *   Все запущенные экземпляры Scheduler'а договариваются между собой, кто будет лидером.
    *   **Лидер** выполняет всю работу (принимает решения о размещении Pod'ов).
    *   Остальные экземпляры **просто ждут** в режиме standby.
    *   Если лидер падает, остальные экземпляры immediately проводят новые "выборы", и один из них становится новым лидером.
    *   Это предотвращает конфликты (например, чтобы два Scheduler'а не попытались разместить один и тот же Pod на двух разных нодах).
