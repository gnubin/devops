
**Inode (Index Node)** —  **метаданные** файла (то есть информация о файле). 

**Важный момент:** inode хранит всё о файле, **кроме двух вещей**:
1.  **Имя файла** (это хранится в директории, которая является просто списком имен и соответствующих им inode).
2.  **Содержимое файла** (это хранится в данных на диске).

### Что хранится в inode?
*   **Размер файла** в байтах.
*   **Права доступа** (кто может читать, писать, запускать файл).
*   **Владелец** файла (User ID) и **группа** (Group ID).
*   **Временные метки** (time stamps):
    *   `atime` — время последнего доступа к файлу (чтение).
    *   `mtime` — время последнего изменения *содержимого* файла.
    *   `ctime` — время последнего изменения *inode* самого файла (например, прав доступа или владельца).
*   **Информация о размещении** на диске — указатели на блоки диска, где физически лежат данные файла. Это самая важная часть, именно по ним система находит содержимое файла.

### Как посмотреть inode файла?

Команда `ls` с флагом `-i`:

```bash
ls -i myfile.txt
1234567 myfile.txt
```
Здесь `1234567` — это номер inode для файла `myfile.txt`.

Более подробную информацию из inode можно посмотреть командой `stat`:

```bash
stat myfile.txt
```
Она выведет почти всё, что хранится в inode.

### Исчерпание inodes: как это происходит и чем грозит

Файловая система при своем создании (командой `mkfs`) резервирует фиксированное количество inodes. 

**Как можно исчерпать inodes?**
Неважно, насколько велики сами файлы. Важно их *количество*. Закончатся inodes, а не дисковое пространство.

Классический пример — сервер, который получает много мелких файлов:
*   **Почтовый сервер** (например, Postfix или Sendmail): каждое письмо — это маленький файл. Миллионы писем = миллионы inodes.
*   **Веб-сервер** с большим количеством маленьких картинок, CSS, JS-файлов.
*   **Кэширующие прокси-серверы** (например, Squid).
*   **Системы с большим количеством процессов**, так как в `/proc` и `/dev` тоже создаются специальные файлы, использующие inodes.
*   **Неудаленные файлы, которые держат открытыми процессы**. Inode освобождается только когда на файл нет ссылок и его не используют процессы.

**Признаки и последствия исчерпания inodes:**

Система будет вести себя так, будто диск полностью заполнен, **даже если команда `df -h` показывает, что свободное место есть!**

1.  **Невозможность создания новых файлов:** Вы увидите ошибку `No space left on device`, хотя место есть. Это самая частая и пугающая ошибка.
2.  **Невозможность создания каталогов:** Директория — это тоже файл, которому нужен свой inode.
3.  **Невозможность запуска программ:** Некоторые программы при запуске создают временные файлы (lock-файлы, pid-файлы). Если inodes нет, программа не запустится.
4.  **Крайняя нестабильность системы:** Многие системные процессы не смогут работать, так как не могут вести свои логи или создавать временные файлы. Система может лечь.

### Как проверить использование inodes?

Используйте команду `df` с флагом `-i`:

```bash
df -i
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/sda1      6553600 6553600      0  100% /
```
Здесь видно, что inodes использованы на 100% (`IUse%`), и свободных (`IFree`) не осталось. При этом `df -h` мог бы показать, что свободное дисковое пространство еще есть.

### Как бороться с проблемой?

1.  **Найти источник проблемы:** Найдите директории с огромным количеством файлов.
    ```bash
    # Показать топ-5 директорий по количеству файлов (может работать медленно)
    sudo find / -xdev -type f | cut -d "/" -f 2 | sort | uniq -c | sort -n | tail -5
    ```
    Или проще:
    ```bash
    # Посчитать количество файлов в конкретной директории (и во всех вложенных)
    sudo find /path/to/directory -type f | wc -l
    ```

2.  **Удалить ненужные файлы:** Чаще всего это старые логи, кэши, временные файлы, содержимое `/tmp`, бесхозные файлы в почтовых очередях.

3.  **Профилактика:**
    *   При создании файловой системы (если это возможно) можно задать большее количество inodes (например, с помощью `mkfs.ext4 -N 10000000 /dev/sdX`), но это не всегда хорошая практика.
    *   Своевременно чистить логи с помощью механизма **logrotate**.
    *   Следить за количеством файлов так же, как и за занятым местом.
    *   Для хранения миллионов мелких файлов иногда лучше использовать специализированные СУБД или объектные хранилища, а не классическую файловую систему.
